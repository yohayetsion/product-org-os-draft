What's New in Product Org OS v5.0.0

Release v5.0.0 Target date May 2026 (Phase 3 Day 10) License MIT (framework) + CC-BY 4.0 (Standard text)

A standard alignment release

Product Org OS v5.0.0 is a different kind of release. Most version bumps add features, polish flows, or harden runtime, and we have done plenty of that across the v3 and v4 lines. This one is about something else. v5.0.0 ships the runnable substrate behind a single, load-bearing claim: that the OS implements the Decision Provenance Standard's reporter protocol at Conformance Level 3.

If you have been following the Vision to Value project, that sentence is the technical surface of a much larger story. The Standard arrived in early 2026 as the open reference document for decision provenance, how organizations record the way decisions are actually made, in a form that auditors, counsel, and regulators can use as input when their work needs that record. The Standard sits in a five-artifact ecosystem: itself, the Vision to Value book (Volume One in the Executive Operating System Series), the Position Paper, the Manifesto Companion, and Product Org OS. Each artifact does a distinct job; this release closes the loop on the OS's job, which is to ship the runnable substrate that lets a Charter authored using the OS achieve Conformance Level 3 when the deployer's discipline matches.

That word, deployer-side, does load-bearing work in this release. The OS provides structural support for Level 3 conformance. Conformance itself is an achievement on the deployer's side, not a feature of the OS. We say that explicitly because the Standard says it explicitly: Section 6 of the Standard grades the Charter; the Standard does not certify. v5.0.0 builds the structural support; you build the discipline; the Standard provides the grading criteria. Each side does its part.

Why a major version

The version jump from v4.0.1 to v5.0.0, skipping the v4.x release line entirely, is the cleanest signal we could give you that this is not a feature drop. A point release on the 4.x line would have undersold what shipped: a new top-level directory carrying eight bundle artifacts, four mode-drift mitigation layers, the Reporter API with an OpenAPI 3.1 wire contract, the Layer 4 attestation schema with eight required sub-fields and four jurisdictional variants, and the Stream E test apparatus that defends the marketing claim. The Standard alignment is a major architectural commitment; the version stamp matches.

If you are upgrading from v4.x, the practical impact is small. The eight named structural-primitive skills retain their existing invocation syntax. The alignment is internal, in the form of new metadata blocks and conformance signal emission. The standard/v5.0/ directory is additive; nothing was removed, nothing was renamed in your daily workflow. End users do not see breaking changes.

Deployers integrating with the Conformance Reporter API see a new wire contract. There is no prior version, so there is nothing to migrate from. The OpenAPI spec ships at version 1.1.0 (an additive enum extension over the originally-locked v1.0 contract closed a documented inconsistency between Layer 3 prose and the OpenAPI binding), and it is backward-compatible by design.

What ships in the alignment bundle

The new directory is standard/v5.0/, and it carries the runnable substrate. Two state machines define the lifecycles. The Charter machine has five forward-only states (open to mode-declared to fields-required to fields-completed to closed), with a review-required interrupt for cases where a record needs to pause and be reviewed before the lifecycle continues. The decision-record machine is the lighter sibling: dispatched / drafted / closed plus the same review-required interrupt. Together they define the operational shape of every decision and Charter the OS produces.

Schemas hold the data. Five required Article 50 disclosure metadata fields ship as a JSON Schema: the declaring authority, the AI system identity, the jurisdictional applicability, a pointer to the disclosure text, and the timestamp the disclosure was attached. Three optional fields cover edge cases (Mode 1 edge case flag, last-reviewed timestamp, disclosure provenance). Charter and decision-record objects each have their own JSON Schema (Draft 2020-12) so that any tool reading or writing them can validate against the Standard's structural contract.

The Conformance Reporter API is where the OS connects to the deployer's broader ecosystem. POST /v2v/conformance/charter-escalation is the single endpoint. The v1.1 amendment extended the escalation_type enum from eight to nine values to include peer_reviewer_pool_underflow, closing a gap between Layer 3's prose specification and the wire contract. The API uses OAuth 2.0 client_credentials, idempotency keys, sync acknowledgement (no async polling), a standardized error envelope, a 60-per-minute rate limit, and JSON Schema Draft 2020-12 validation. The full OpenAPI 3.1 spec ships with the bundle. A signal vocabulary file documents the named conformance signals across Levels 1, 2, and 3, with sample audit-cadence binding (15% rolling sample with first-100 and edge-case overrides, the cadence calibration the Standard's Section 6 references).

The four-layer Mode-Drift mitigation

This is the piece of v5.0.0 that took the most architectural work. Mode-drift is the risk that a record originally classified as Mode 1 (the lighter-weight classification) silently becomes a Mode 2 record over time without the classification being updated. This matters because Mode 2 records carry stricter disclosure and review obligations; a silent drift creates a gap between the record's actual content and its declared classification.

Four layers compose the mitigation.

Layer 1 is the statistical detection layer. It ships in v5.0.0 as scaffolding only. The classifier is in the bundle, but enforcement gates phase in starting at OS v5.0 lifecycle Week 7+, after the Layer C adversarial corpus is authored in Weeks 4-6. Until then, Layer 1 runs in detection-only mode, sampling at 15% baseline (with overrides for edge cases: 30% for mode-1-with-embedded-mode-2-summary Charters, every-record-until-first-100 for early lifecycle), with a 0.75 hard threshold and a 0.50-0.75 soft-flag band. The three-corpus design (Layer A: 200 Mode-2 records; Layer B: Section 4 + 0.5.D worked examples; Layer C: adversarial) is what you train the classifier against.

Layer 2 is the in-flow audit hook. It fires hot on day one of v5.0.0. Every Mode-1 record close triggers a 4-question Substantive-Authorship Challenge, captured in a 10-field audit trail. This is the layer that catches mode-drift in real time, before it has a chance to ossify in the historical record.

Layer 3 is the Mode-Confirmation Audit primitive. Also hot day-one. When a record needs review, the lifecycle's review-required interrupt activates, a peer reviewer is designated (per a 5-rule designation algorithm: pool minimum of 3, author and Charter-owner exclusions, recusal carve-out), and the record waits for sign-off before continuing. Layer 3 has firing authority for the no_silent_mode_drift_in_sample Level 2 conformance signal on Seam 3 cadence. That is, this layer is what enforces the Level 2 grading criterion at runtime.

Layer 4 is the named human-attestation layer. Hot day-one. The mode_classification_attestation is a structured object with eight required sub-fields (full name, role, employer, timestamp, jurisdiction enum, language version, verbatim signed text, capacity), and verbatim attestation language in four jurisdictional variants (United States, United Kingdom, European Union, Israel). General Counsel was load-bearing in the Layer 4 design. The structured-object form (rather than a scalar attestation field) was a deliberate refinement to make the attestation traceable, the jurisdictional variant explicit, and the verbatim signed text an immutable audit-trail element.

Together, Layers 2-4 fire hot on day one. Layer 1 is scaffolding-only at launch and activates as the lifecycle runs. R-001 (silent Mode 1 to Mode 2 drift) closes day one because the three load-bearing layers run hot from the start.

The eight named structural-primitive skills

These are the OS skills whose alignment to the Standard is load-bearing for the Conformance Level 3 claim.

/decision-charter is the most directly aligned. Its output schema matches Standard Section 3 exactly: the 5-state lifecycle, the 15 required Charter fields, the 3-value mode-declaration enum. /decision-record aligns to Standard Section 5 and adds two new fields (dispatch_mode and disclosure_metadata_pointer) so that every decision record can trace back to the Charter that authorized it and the disclosure metadata that documents how it was disclosed. /escalation-rule integrates with Layer 3's Mode-Confirmation Audit primitive and is what fires when an escalation routes through the lifecycle.

The other five (/ownership-map, /customer-value-trace, /collaboration-check, /scale-check, /phase-check) emit conformance signals at Levels 1 and 2 where applicable. /ownership-map emits the accountable_owner_named Level 1 signal. The others emit Level 2 signal stubs that document the structural property each skill validates. The signals flow through the Reporter API to whatever conformance tracking system the deployer maintains.

The remaining ~125 ancillary OS skills are not load-bearing for the Conformance Level 3 claim. They receive Mode-awareness and disclosure-metadata field alignment in v5.1 (4-6 weeks post-launch). v5.1 improves coverage breadth, not conformance depth, meaning that even today, with only the eight named skills aligned, a Charter authored using v5.0.0 OS can achieve Level 3 when deployer-side discipline matches. v5.1 means more of your daily skills emit signals automatically; v5.0.0 means the eight skills that matter for Level 3 already do.

The Standard alignment, in one sentence

If you read just one paragraph from this page, read this one.

Product Org OS v5.0 implements the Decision Provenance Standard's reporter protocol at Conformance Level 3.

That sentence is the marketing-strong canonical claim, locked verbatim in the Conformance Framing Locked Language Table. Use it on technical surfaces. The lay-audience paraphrase, for non-technical readers (engineered to meet the Decision Provenance Standard's highest current conformance level), is an authorized variant scoped to LinkedIn day-one and press-release lay paragraphs.

What the canonical claim does not say is also load-bearing. It does not say the OS is "compliant" with the Standard, nor "certified," nor "regulator-ready," nor "audit-defensible." Those phrasings are forbidden across all v5.0.0 surfaces because they overclaim. The Standard does not confer compliance authority, there is no certification track, and no regulator has endorsed the Standard. What the canonical claim does say (implements the reporter protocol at Conformance Level 3) is precise: the OS ships the structural mechanism the Standard's Section 6 grades against at Level 3. Whether your specific deployment achieves Level 3 depends on you.

What you do next

If you are already using Product Org OS, the upgrade is straightforward. git pull the new v5.0.0 tag, run python install.py to refresh your local skills directory, and your eight named structural-primitive skills will pick up the new metadata block automatically. The standard/v5.0/ directory is now visible in the repo root. Explore it at your own pace. Nothing in your existing workflow breaks.

If you are integrating with the Conformance Reporter API as a deployer, the OpenAPI 3.1 spec at standard/v5.0/conformance/reporter-api.openapi.yaml is the contract. The reference implementation pattern is documented in standard/v5.0/conformance/reporter-api-spec.md. Where you host the reporter endpoint is your call. The recommended pattern is deployer-hosted at a domain you control, with the OS as the client.

If you are new to the Vision to Value ecosystem, the Standard hub at the canonical URL (live at decisionprovenance.ai if domain attachment lands by Day 5-6 of Phase 3.5; fallback at executiveoperatingsystems.org/standard/ if not) is the single best entry point. From there you can read the Standard's full text, download the schemas, find the citation guidance (the Standard is licensed under CC-BY 4.0 with attribution), and see the list of conforming tools and organizations. The list is empty at launch by design. It grows as deployers register their conforming Charters. Yours could be one of the first.

If you want the broader story, Vision to Value: Volume One in the Executive Operating System Series is the book that frames everything you have just read. Volume One is the CPO's seat. The series is open-author by design. Other volumes covering the executive seats whose decision authority the same organizational control architecture covers are invited contributions from authors with the lived experience to write them.

The release calendar

v5.0.0 ships at the Phase 3 Day 10 publish target. Day 8 is code freeze. Day 9 is Stream E documentation integration. Day 10 is tag v5.0.0, publish GitHub release, update marketing site, deploy this whats-new-v5.0.0 page.
